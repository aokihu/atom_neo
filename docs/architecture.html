Key difference from v1: No polling loop. Task submission fires an event that the Task Engine consumes immediately. All modules communicate exclusively through the Pipeline Event Bus (single source of truth for events).

3. Event-Driven Scheduler

sequenceDiagram participant C as Client participant WS as Core::WS participant BUS as Event Bus participant TE as Task Engine participant PIPE as Pipeline C->>WS: task.submit WS->>BUS: emit("task.enqueued", task) BUS->>TE: onTaskEnqueued(task) alt No task running TE->>BUS: emit("task.activated", task) BUS->>PIPE: runPipeline(task) else Task running TE->>TE: queue task end PIPE-->>BUS: emit("pipeline.element.finished") BUS-->>WS: broadcast to client PIPE-->>BUS: emit("pipeline.finished", result) BUS->>TE: onPipelineFinished(result) TE->>TE: dequeue next task (if any)
v1 Equivalent: while(true) { if (queue.isEmpty) await sleep(500); }. The polling loop is removed entirely. Task Engine subscribes to bus events and reacts. No timers, no wasted CPU.

4. Pipeline Builder DSL

v1 — Hardcoded

return {
  elements: [
    new CollectPrompts({ ctx, runtime }),
    new FormatMessages({ ctx, runtime, transportConfig }),
    new StreamLLM(ctx, serviceManager),
    // ... 7 more
  ],
};

v2 — Declarative Builder NEW

pipeline("conversation")
  .source("collect-prompts", { runtime })
  .transform("format-messages", { runtime, config })
  .transform("stream-llm", { serviceManager, tools, bus })  // streamText + tool calling
  .boundary("check-follow-up")  // parse follow_up IntentRequest
  .sink("finalize", { runtime })
  .build();

Element Registry

const elements = new Map();
elements.set("collect-prompts", CollectPromptsElement);
elements.set("stream-llm", StreamLLMElement);
elements.set("check-follow-up", CheckFollowUpElement);
elements.set("finalize", FinalizeConversationElement);
